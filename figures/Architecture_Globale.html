Le service central : AiContentService

C'est ici que toute la logique IA est concentrée. Il implémente :

  • GeneratePostAsync — génère un post unique
  • SuggestCampaignAsync — pipeline 3 étapes (strategy → planning → campaign)
  • GenerateCampaignStrategyStepAsync / GenerateCampaignPlanningStepAsync / GenerateCampaignContentStepAsync — les 3 étapes exposées séparément pour le stepper interactif
  • MaterializeCampaignAsync — transforme une suggestion IA en vrais objets en base
  • ChatWithAssistantAsync — conversation avec l'assistant

Comment les agents IA sont intégrés — le chemin complet

4.1 L'interface dans Application

backend/AiContentFlow.Application/Common/Interfaces/ILocalAiBackendClient.cs — ligne 1

public interface ILocalAiBackendClient
{
    Task<JsonElement> GenerateContentAsync(...)     // → Orchestrator Agent
    Task<JsonElement> GenerateStrategyAsync(...)    // → Strategy Agent
    Task<JsonElement> GeneratePlanningAsync(...)    // → Planning Agent
    Task<JsonElement> GenerateCampaignContentAsync(...) // → Campaign Agent
    Task<JsonElement> AssistantChatAsync(...)       // → Assistant Agent
    Task<JsonElement> AnalyzeBrandAsync(...)        // → Brand Agent
    Task<JsonElement> GeneratePosterAsync(...)      // → Creative Agent (poster)
    Task<JsonElement> GenerateCarouselAsync(...)    // → Creative Agent (carousel)
}
Cette interface est dans la couche Application. Elle ne sait pas comment les agents sont implémentés.

4.2 L'implémentation dans Infrastructure

backend/AiContentFlow.Infrastructure/AI/LocalAiBackendClient.cs — ligne 250

LocalAiBackendClient est un client HTTP qui appelle le backend Python. Chaque méthode de l'interface correspond à un endpoint Python :

Méthode C#Endpoint PythonAgent
GenerateContentAsyncPOST /api/orchestrator/runOrchestrator Agent
GenerateStrategyAsyncPOST /api/strategy/generateStrategy Agent
GeneratePlanningAsyncPOST /api/planning/generatePlanning Agent
GenerateCampaignContentAsyncPOST /api/campaign-content/generateCampaign Agent
AssistantChatAsyncPOST /assistant/chatAssistant Agent
AnalyzeBrandAsyncPOST /api/brand/analyzeBrand Agent
GeneratePosterAsyncPOST /api/creative/generate-posterCreative Agent

La méthode SendAsync interne (ligne 250) gère :

  • Circuit breaker : si 4 échecs consécutifs → circuit ouvert 45 secondes
  • Retry : 2 tentatives avec délai exponentiel
  • Timeouts différenciés : 75s par défaut, 95s pour brand analyze, 900s pour campaign content (génération de posts multiples avec images)
  • X-Correlation-ID : chaque appel transporte un ID de corrélation pour le tracing

4.3 Le pipeline 3 étapes pour les campagnes

Dans AiContentService.SuggestCampaignAsync (ligne 206) :

1

Strategy Agent

GenerateStrategyAsync(goal, theme, platforms)
→ retourne { strategy_id, summary, content_direction[] }

2

Planning Agent

GeneratePlanningAsync(strategy, strategyId, postsPerWeek)
→ retourne { planning_id, summary, weeks[{ days[] }] }

3

Campaign Agent

GenerateCampaignContentAsync(strategy, planning, planningId, brandContext)
→ retourne { campaign_name, weeks[{ days[{ generated_content }] }] }

Chaque étape reçoit le résultat de la précédente en entrée. En cas d'échec à n'importe quelle étape, un fallback avec 3 posts génériques est créé (BuildFallbackCampaign).

4.4 Le mode External vs LocalBackend

Le service supporte deux modes configurables par AI:ProviderMode :

Mode LocalBackend (défaut)

Appels via le backend Python

Appelle le backend Python local via LocalAiBackendClient.

Mode ExternalProviders

Appels directs aux APIs IA

Court-circuite le backend Python et appelle directement TextGenerationService, qui lui-même gère la chaîne de fallback.

// TextGenerationService.cs ligne 117
// Plan de fallback des providers :
AiUseCase.GeneratePost    → Groq → OpenAI → Gemini
AiUseCase.BrandExtraction → Gemini → Groq → OpenAI
AiUseCase.SuggestCampaign → Groq → OpenAI → Gemini

TextGenerationService appelle directement les APIs externes :

  • CallGroq() → api.groq.com/openai/v1/chat/completions avec llama-3.3-70b-versatile
  • CallOpenAI() → api.openai.com/v1/chat/completions avec gpt-4o-mini
  • CallGemini() → generativelanguage.googleapis.com avec gemini-2.0-flash
  • CallDeepSeek() → api.deepseek.com/v1/chat/completions

Le LinkedInPublisher en détail

backend/AiContentFlow.Infrastructure/Publishers/LinkedInPublisher.cs — ligne 31

  1. Extrait le texte du ContentJson du PostVariant
  2. Si image présente → UploadImageAsync : register upload → upload binaire → récupère l'assetUrn
  3. Appelle POST api.linkedin.com/v2/ugcPosts avec l'UGC format
  4. Fallback automatique : si urn:li:member:{id} échoue → retry avec urn:li:person:{id}
  5. Retourne PublishResult.Success(postId, url) ou PublishResult.Failure(message)

Le cycle de vie complet d'une publication

1

Déclenchement utilisateur

Utilisateur clique "Publish" dans React → POST /api/teams/{id}/publish-jobs
→ PublishJobsController → PublicationService.ScheduleAsync()

2

Création du job en base

PublishJob créé en base (status: Pending, ScheduledAt = now)

3

Hangfire — chaque minute

PublishScheduledVariantsJob.ExecuteAsync() → ClaimDueAsync() → lock optimiste sur les jobs à traiter

4

Traitement de chaque job

a. Récupère SocialAccount + vérifie token valide
b. Récupère PostVariant + merge image (VariantContentMerge)
c. PublisherFactory.GetPublisher(platform)
d. LinkedInPublisher.PublishAsync(variant, account)

5

Appel LinkedIn

POST api.linkedin.com/v2/ugcPosts → retourne postId

6

Mise à jour des statuts

publication.MarkPublished(postId, url, DateTime.UtcNow)
→ SyncContentPostPublishedStatusAsync() met à jour le ContentPost

7

Collecte des analytics — chaque heure

SyncPublicationAnalyticsJob.ExecuteAsync() → InsightsProviderFactory.GetProvider(platform) → LinkedInInsightsProvider.FetchPostInsightsAsync()
→ Sauvegarde PublicationAnalytics avec déduplication SHA256

L'Orchestrator Agent coordonne séquentiellement un ensemble d'agents spécialisés, accessibles via une intégration API externe

Pour les campagnes complexes, un Orchestrator Agent coordonne séquentiellement un ensemble d'agents spécialisés, accessibles via une intégration API externe. Le tableau 7 présente ces agents et leurs rôles.

Ce que cette phrase signifie concrètement dans le code — chaque élément ancré dans le code réel.

Deux niveaux d'orchestration superposés

Il y a deux niveaux d'orchestration superposés. C'est la clé pour tout comprendre.

Niveau 1 — Orchestrateur côté .NET

AiContentService

Fichier : backend/AiContentFlow.Application/Features/Ai/AiContentService.cs
AiContentService est l'orchestrateur .NET. Il décide de l'ordre d'appel des agents, transmet les résultats d'un agent au suivant, et gère les erreurs avec fallback. Il joue le rôle de chef d'orchestre côté backend .NET.

Niveau 2 — Backend Python

http://127.0.0.1:8000

C'est là que les agents spécialisés vivent réellement. Chaque endpoint Python est un agent autonome. Le "Orchestrator Agent" du rapport correspond à l'endpoint /api/orchestrator/run côté Python, pour la génération de post simple. Pour les campagnes, c'est AiContentService lui-même qui joue ce rôle d'orchestrateur.

Le cas du Orchestrator Agent — génération de post simple

Pour un post simple (Quick Generate), c'est l'endpoint /api/orchestrator/run qui est appelé directement. L'orchestrateur Python prend la décision interne de mobiliser ou non le Creative Agent selon le paramètre needs_creative.

Dans LocalAiBackendClient.GenerateContentAsync (ligne 105), le corps envoyé au Python est :

{
  "mode": "manual_brand",        ← "prompt_only" | "manual_brand" | "scraped_brand"
  "org_id": "team_abc123",       ← identifiant de marque dans le Python
  "prompt": "Écris un post LinkedIn sur...",
  "platforms": ["linkedin"],
  "language": "French",
  "brand_context": {             ← contexte de marque complet
    "brand_name": "Consultim-IT",
    "tone_of_voice": ["professionnel", "inspirant"],
    "content_pillars": ["digital", "cloud", "IA"],
    ...
  },
  "content_type": "social_post",
  "post_type": "text_post",
  "type": "linkedin_standard",
  "needs_creative": false        ← si true → Python mobilise Creative Agent
}

Le mode (prompt_only, manual_brand, scraped_brand) indique à l'Orchestrator Python quelle source de contexte de marque utiliser.

La synchronisation de marque — Brand Agent avant les campagnes

Avant tout appel de campagne, AiContentService appelle SyncBrandToAiInternalAsync qui envoie le profil complet de marque au backend Python via POST /api/brand/manual. C'est LocalAiManualBrandMapper.ToApiBody() (ligne 11) qui construit ce payload, avec les champs parsed_profile et enriched_profile — des dizaines de champs incluant archétype de marque, guidelines de voix, couleurs visuelles, etc.

Le Python stocke cette marque sous orgId et la réutilise dans tous les appels de stratégie et de contenu suivants.

Résumé en une phrase pour le jury

AiContentService (.NET, couche Application) joue le rôle d'orchestrateur en appelant séquentiellement les endpoints REST du backend Python, chacun représentant un agent spécialisé. LocalAiBackendClient (.NET, couche Infrastructure) implémente l'intégration API externe avec circuit breaker, retry et timeouts différenciés. La couche Domain ne voit jamais l'IA — elle reçoit uniquement les entités ContentPost et Campaign créées par matérialisation.
