La philosophie Feature-Sliced Design (FSD)

Chaque feature est un dossier autonome qui suit toujours la même structure interne :

features/campaigns/
├── campaigns.api.ts ← fonctions d'appel HTTP (fetch/POST vers le backend)
├── campaigns.queries.ts ← hooks TanStack Query (useQuery/useMutation)
├── campaigns.types.ts ← types TypeScript
├── campaigns.display.ts ← utilitaires d'affichage (labels, couleurs...)
├── components/ ← composants React de cette feature
│ ├── CampaignCard.tsx
│ ├── CampaignAiStepper.tsx
│ └── ...
├── pages/ ← pages complètes (une page = une route)
│ ├── CampaignAiPlanPage.tsx
│ ├── CampaignOverviewPage.tsx
│ └── ...
├── hooks/ ← hooks métier propres à la feature
│ ├── useCampaignAiPipeline.ts
│ └── useCampaignContext.ts
├── layouts/ ← layouts avec Outlet React Router
│ └── CampaignWorkspaceLayout.tsx
└── lib/ ← helpers spécifiques
    ├── campaignAi.types.ts
    └── formatGeneratedContent.ts
Cette règle est identique pour channels/, auth/, posts/, analytics/, etc.

La couche App — frontend/src/app-v2/app/

Providers (providers.tsx)

<StyledEngineProvider injectFirst>        // MUI — priorité CSS
  <ThemeProvider theme={appTheme}>        // thème global Material UI
    <CssBaseline />                        // reset CSS
    <QueryClientProvider client={queryClient}> // TanStack Query (cache HTTP)
      {children}
    </QueryClientProvider>
  </ThemeProvider>
</StyledEngineProvider>

Le QueryClient est configuré avec :

  • staleTime: 60_000 → les données restent fraîches 60 secondes (pas de refetch inutile)
  • refetchOnWindowFocus: false → pas de rechargement quand l'utilisateur revient sur l'onglet

Router (AppV2Router.tsx)

La structure de routage reflète exactement l'architecture métier :

/login, /register, /forgot-password      ← PublicOnlyRoute (redirige vers /dashboard si connecté)
/dashboard                               ← DashboardPage
/generate                                ← EditorRoute → GeneratePage (Admin/Editor uniquement)
/content-feed                            ← ContentFeedPage
/calendar                                ← CalendarPage
/channels                                ← ChannelsListPage
/channels/:channelId/                    ← ChannelWorkspaceLayout (layout imbriqué)
  overview, campaigns, content, publishing, analytics, settings
  posts/new, posts/:postId               ← PostEditorPage
/channels/:channelId/campaigns/:campaignId/  ← CampaignWorkspaceLayout
  overview, posts, timeline, analytics, settings
/channels/:channelId/campaigns/:campaignId/ai-plan  ← CampaignAiPlanPage
/campaigns                               ← CampaignsListPage (vue cross-channel)
/integrations/social-accounts            ← SocialAccountsPage
/brand-studio                            ← BrandStudioPage
/invite-user                             ← AdminRoute (Admin uniquement)
/profile                                 ← ProfilePage

Les routes gardées sont des composants Outlet spécialisés :

  • ProtectedRoute → redirige vers /login si pas de token JWT
  • PublicOnlyRoute → redirige vers /dashboard si déjà connecté
  • AdminRoute → bloque si rôle ≠ Admin
  • EditorRoute → bloque si rôle = Viewer

La couche Shared — le client HTTP

http.ts — frontend/src/app-v2/shared/lib/http.ts

C'est le fichier central qui gère toute la communication avec le backend. Il est basé sur Axios avec deux intercepteurs :

Intercepteur requête

Ligne 17 — Attache le JWT Bearer

// Attache automatiquement le JWT Bearer à chaque requête
config.headers.Authorization = `Bearer ${token}`;
Intercepteur réponse

Ligne 26 — Gestion automatique du refresh token

Réponse 401 (token expiré) ?
    → appelle POST /Auth/refresh avec le refreshToken
    → si succès → relance la requête originale avec le nouveau token
    → si échec  → vide le localStorage + redirige vers /login
Mécanisme de lock

Ligne 132 — Prévention des refreshs multiples simultanés

Si plusieurs requêtes expirent en même temps, un seul refresh est exécuté grâce à navigator.locks.request(). Les autres attendent le résultat du premier.

storage.ts — frontend/src/app-v2/shared/lib/storage.ts

Centralise tout ce qui est stocké en localStorage :

CléContenu
accessTokenJWT d'accès (courte durée)
refreshTokenToken de refresh (longue durée)
userIdID de l'utilisateur connecté
usernameNom d'affichage
teamIdteamId de l'équipe active → utilisé dans toutes les URLs API
teamRoleAdmin / Editor / Viewer → RBAC frontend
emailEmail de l'utilisateur

Couche Features — comment chaque feature est organisée

Pattern .api.ts → .queries.ts → composant

Niveau 1

campaigns.api.ts — fonctions pures qui appellent l'API

// campaigns.api.ts
export async function getCampaigns(channelId: number) {
  const teamId = authStorage.getTeamId();
  return apiRequest<Campaign[]>(`/teams/${teamId}/channels/${channelId}/campaigns`);
}
Niveau 2

campaigns.queries.ts — hooks TanStack Query

// campaigns.queries.ts
export function useCampaigns(channelId: number) {
  return useQuery({
    queryKey: ["campaigns", channelId],
    queryFn: () => getCampaigns(channelId),
  });
}
Niveau 3

Composant / page — consomme le hook

// CampaignsListPage.tsx
const { data: campaigns, isLoading } = useCampaigns(channelId);
Ce pattern est identique dans toutes les features : auth, channels, analytics, brand-studio, ai, etc.

La feature IA — comment le pipeline agents est orchestré côté frontend

ai.api.ts — frontend/src/app-v2/features/ai/ai.api.ts

Définit les fonctions d'appel pour chaque étape du pipeline et les types TypeScript correspondants. Points importants :

// Timeout côté client : 16 minutes (dépasse le backend 15min)
const CAMPAIGN_CONTENT_CLIENT_TIMEOUT_MS = 16 * 60 * 1000;

// Chaque étape est un appel API distinct
generateCampaignStrategy(config)   → POST /teams/{id}/ai/campaigns/strategy
generateCampaignPlanning(payload)  → POST /teams/{id}/ai/campaigns/planning
generateCampaignContent(payload)   → POST /teams/{id}/ai/campaigns/content

useCampaignAiPipeline.ts — frontend/src/app-v2/features/campaigns/hooks/useCampaignAiPipeline.ts

C'est le cerveau du workflow IA côté frontend. C'est un hook custom qui gère l'état de toutes les étapes :

État interne géré

activeStepétape courante (0=configure, 1=strategy, 2=planning, 3=content, 4=confirm)
strategyProfilerésultat JSON de l'étape Strategy Agent
strategyIdID retourné par le backend
selectedDirectiondirection de contenu choisie par l'utilisateur
planningProfilerésultat JSON de l'étape Planning Agent
planningIdID retourné par le backend
postsliste des posts générés (étape Campaign Agent)
busyloading state global
errormessage d'erreur formaté

Fonctions exposées

  • runStrategy() → appelle generateCampaignStrategy() → stocke strategyProfile
  • runPlanning() → appelle generateCampaignPlanning(strategyProfile) → stocke planningProfile
  • runContent() → appelle generateCampaignContent(strategyProfile + planningProfile)
  • finalize() → appelle bulkCreateCampaignPosts() pour sauvegarder en base
L'utilisateur peut modifier les résultats de chaque étape avant de passer à la suivante (ex : éditer la stratégie générée avant de lancer le planning).

La gestion des permissions RBAC côté frontend

useTeamPermissions.ts — frontend/src/app-v2/shared/hooks/useTeamPermissions.ts

const role = authStorage.getTeamRole();  // "Admin" | "Editor" | "Viewer"

return {
  isAdmin:          role === "Admin",
  isEditor:         role === "Editor",
  canMutateContent: isAdmin || isEditor,  // peut créer/modifier du contenu
  canManageTeam:    isAdmin,               // peut inviter des membres
  canImportBrand:   isAdmin,               // peut importer une marque
}

Utilisé partout dans les pages :

// CampaignAiPlanPage.tsx ligne 89
if (!canMutateContent) {
  return <Alert>You need Editor or Admin permissions to use AI planning.</Alert>
}
Important : La vérification est doublée côté backend (RBAC dans AiContentService) — le frontend est une garde UI, pas une protection de sécurité.

L'autosave de l'éditeur de post

usePostAutosave.ts — frontend/src/app-v2/features/posts/hooks/usePostAutosave.ts

Implémente une sauvegarde automatique intelligente :

①

Saisie utilisateur

L'utilisateur tape dans l'éditeur → isDirty = true

②

Debounce 2 secondes (DEBOUNCE_MS)

Si l'utilisateur arrête de taper 2s → runSave()

③

Periodic check toutes les 30 secondes (PERIODIC_MS)

Si isDirty → runSave() (filet de sécurité)

④

runSave() — logique de sauvegarde

Si post nouveau → onCreate() → reçoit l'ID créé
Si post existant → onUpdate()
setSaveState("saving" → "saved" | "error")

⑤

Protection contre la concurrence

savingRef.current empêche deux saves simultanés.
